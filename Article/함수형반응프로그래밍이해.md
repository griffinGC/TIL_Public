# 함수형 리액티브 프로그래밍 이해하기

> https://itnext.io/demystifying-functional-reactive-programming-67767dbe520b

지난 몇년동안 리액티브 프로그래밍은 많은 관심을 받아오는 중이다. [reactive manifesto](https://www.reactivemanifesto.org/) 는

반응형 소프트웨어 설계에 대해 논의하고 수천번 서명해왔다. 현재 ReactiveX 같은 반응형 라이브러리는 기본적으로 모든 주요한 프로그래밍 언어에 존재한다.

빅데이터와 Spark의 출현으로, 함수형 프로그래밍 역시 꽤 과장된 광고를 경험하고 있다. 더한 혼란을 야기하면서, *functional reactive programming* 은 섞이게 되고 빈번하게 ReactiveX 같은 반응형 라이브러리를 설명하는데사용된다.

이 아티클에서는 모든 유행어에 대해 다룰 것이다. 즉 함수형 프로그래밍, 반응형 프로그래밍, 함수형 리액티브 프로그래밍이 무엇인지, 어떻게 서로 관련이 있고 왜 알아야 하는지?

*(스칼라 스티펫를 이용하여 개념을 설명을 할 예정이다. 스칼라를 잘 몰라도 쉽게 따라올 수 있도록 간단하게 사용할 것이다.)*



## 함수형 프로그래밍

함수형 프로그래밍은 모두 함수에 관한 것이다. 음, 사실 이건 모두 pure 함수에 대한것이다.

Pure 함수는 어떠한 사이드 이펙트도 없는 함수이다. 다른 말로 하자면, 이것은 데이터 베이스에 작성할 수 없다. 이것은 이것의 스코프 밖에서 어떠한 값도 변경할 수 없고, 어떠한 것도 출력할 수 없다.

이건 아주 중요한 결과이다. Pure 함수는 항상 무언가를 리턴해야 한다. 어떠한 상태도 변경하지 않고 아무것도 리턴하지 않는 함수는 쓸모 없다.

보라.

```scala
def useless_function_times_two(x: Int): Unit = {
    val useless_result = x * 2
} 
```

명백하게, 이 함수는 아무 쓸모도 없다. (이 경우에 이건 명백하지 않다. 왜냐하면 스칼라에 대해 모르기 때문이다. 함수는 이것의 결과를 리턴하지 않고 어떠한 외부 값도 변경하지 않는다.)



### 참조 투명성

또한, pure 함수는 오직 인자로써 함수에 넘겨진 인자에만 의존한다. 이것은 외부에서 안으로 어떤것에도 접근할 수 없다. 

한가지 중요한 이점이 있다. *참조 투명성* . 같은 인자를 가진 pure 함수를 부를때, 이것은 항상 같은 결과를 리턴한다.

이것은 코드에 대한 추론을 훨씬 쉽게 할 수 있다. 만약 각각의 함수에 대해 테스트 하고, 그것들이 동작하는것에 확신이 있다면, 서로 상호작용하는 그 함수들이 너에게 너가 예상하는 결과를 준다는 확신 역시 있어야 한다.

이어지는 impure 함수를 보라.

```scala
var value = 2
def value_times_two(): Int = value * 2 // Impure function - bad idea
```

만약 a) `value` 가 의존하는게 무엇인지 b) `value`의 값이 무엇인지 모른다면 이 함수가 무엇을 리턴할지 예측할 방법이 없다. 

이 예제가 약간 부자연스럽지만, 아마도 모든 노련한 프로그래머들은 이미  비슷한 상황에 놓인적이 있었을 것이다. 그리고 리턴 값이 어디서 오는지 궁금해 했을 것이다. 대부분의 함수는 위에 있는 것보다 약간 더 복잡하다.  만약 그 함수들이 외부 의존성을 가지고 있다면, 이러한 의존성을 변경하는 것은 절대 보이지 않는 결과를 초래할 수 있을 것이다.

아래 있는 것이 일반적으로 더 좋은 생각이다.

```scala
def times_two(x: Int): Int = x * 2
```

함수형 프로그래머는 이게 함수를 작성하는 유일한 올바른 방법이라고 말한다. 그리고 몇몇 함수형 프로그래밍 언어는 impure 함수를 작성하는것을 허용하지 않을 것이다. 너가 하드코어 함수형 프로그래머이든 아니든 간에, 참조 투명성은 항상 목표가 되어야 한다고 생각한다.



### 불변성

동시성은 pure 함수에 대한 다른 요구사항을 이끌었다. 함수에 입력값으로 전달된 변수들은 불변이어야만 한다. 그렇지 않으면 다른 스레드는 함수에 변수를 전달한 이후에 변수의 값을 변경할 수 있다. 그것은 참조투명성을 깨는 것이다.

그런데, 함수의 결과값을 다른 함수의 입력값으로 전달하기를 원하기 때문에 , pure 함수의 리턴 값은 불변이어야 한다.

몇몇 함수형 프로그래밍 언어들은 어떤것도 변경하지 못하도록 한다. 리스트의 값을 2배로 만들고 싶은가? 리스트를 함수로 넘기고 대신 새로운 리스트를 리턴 받아라. 리스트에 대해서 반복하고 리스트의 어디에 있는지 추적하기 위해 변수를 필요로 하는가? 대신 [재귀](https://en.wikipedia.org/wiki/Recursion_(computer_science))를 사용해라.

이것은 동시성과 실패했을때 회복하는 것을 필요로 하는 분산 시스템을 다룰때 상당한 이점이 된다. 만약, 아무것도 변경할 수 없다면, 경쟁 조건(race condition) 이나 데드락에 대해 걱정할 필요가 없다. 또한, 만약 분산 시스템에서 많은 계산을 수행하고 노드들 중 하나가 실패하는 경우, 만약 원본 불변 값을 가지고 있고 수행해야할 연산을 알고 있다면 복구하기 쉽다. (많은 데이터 양을 처리하는 유명한 프레임워크중 하나인 Apache Spark는 이렇게 동작한다.)

많은 함수형 프로그래밍 책이 있고 많은 책들이 이것에 관해 쓰여졌다. 그러나, 이건 기본적인 요지이다.



## 리액티브 프로그래밍

리액티브 프로그래밍은 스프레드 시트와 유사하게 설명된다. 다른 두개의 셀을 계산하는 셀을 생각해봐라. 입력값중 하나를 변경했다면, 합또한 변경된다. 셀은 변화에 *반응* 하고 스스로 업데이트 한다.

이것은 [dataflow programming](https://en.wikipedia.org/wiki/Dataflow_programming)과 비슷하다. 개념적으로, 여기서는 제어의 흐름 대신 데이터의 흐름에 초점을 맞춘다.

리액티브 행위를 구현하는 방법중 하나는 콜백을 제공하는 Futures(혹은 다른 언어의 Promise)를 사용하는 것이다. 예를 들면 아래의 코드 같이 말이다.

```scala
import scala.concurrent.{Future}
import scala.util.{Failure, Success}
import scala.concurrent.ExecutionContext.Implicits.global

val expensive_computation = Future{
    Thread.sleep(1000) // Simulate long-running computation
    2 * 2
}

expensive_computation.onComplete{
    case Success(value) => println(s"Result of expensive computation: $value")
    case Failure(e) => e.printStackTrace
}
```

Futures는 동시에 긴 러닝 코드를 쉽게 실행하고 결과값을 이미 사용가능한 것처럼 처리할 수 있도록 하는 추상화를 제공한다. 일단 코드의 긴 실행조각이 완료되면, 콜백이 수행된다. - 이것은 완료에 *반응*한다.

 Futures가 동시에 실행되어야 하는 간단한것을 처리하기위한 좋은 방법일지라도, 이것은 좀 더 복잡한 것을 수행하길 원한다면 지저분해진다. 예를 들어,  Futures를 이용하여 스프레드 시트 어플리케이션을 구현하는것을 생각해보자.  스칼라의 Futures를 이해하여 다른방식과 결합하고 구성하더라도, 곧 엉망이 될 것이다.

Futures와 반대로,  Rx 같은 가장 인기있는 반응형 라이브러리는 이벤트를 데이터 스트림으로 본다. (또한, 그것들은 'observalbe' 이라고 불리지만, 우리는 더 직관적이기 때문에 'streams'라고 부를 것이다. ) 그것들은 이벤트 스트림이 변경되고 결과 값을 함께 붙이는 것을 가능하게 한다. Futures는 한번만 사용될지라도, 이벤트 스트림은 이벤트가 일어날때마다 액션을 실행하는데 사용될수 있다. 그래서 이벤트 스트림은 Futures와 비슷하지만, 시간의 차원을 추가하기 때문에 더 강렬하다.

이것들을 더 명확하게 해주는 예제를 살펴보자. 유저가 더블 클릭할때마다 이벤트를 수행하고 싶다고 해보자. 이것을 스트림으로 나타내면 다음과 같다.

![stream_visualizing](https://miro.medium.com/max/512/1*FkLrVLyxxN5OH9we63lHiQ.jpeg)

클릭 이벤트를 스트림으로서 볼때, 더블클릭에 대해서 이 스트림을 다른 이벤트의 스트림으로 변환하는 간단한 선언적 API를 사용할 수 있다. 이제 간단히 유저가 더블 클릭을 할때 실행하고 싶어하는 것이 무엇이던간에 실행하는 콜백을 가진 스트림을 구독하면된다.

이 아티클의 스코프를 넘고 내가 할 수 있는 것보다 이미 더 잘 설명한 더 많은 반응형 프로그래밍과  API들이 존재한다. 자바스크립트로 된 Rx의 완벽한 예제를 보고 싶다면 Andre Staltz 의 [Missing introduction to Reactive Programming](https://gist.github.com/staltz/868e7e9bc2a7b8c1f754) 를 확인해라. 혹은 ReactiveX 웹 사이트에 있는 [list of Tutorials](http://reactivex.io/tutorials.html) 를 봐라



### 리액티브 시스템

이제, [reactive manifesto](https://www.reactivemanifesto.org/) 에서 설명한것처럼 어떻게 이게 모두 리액티브 시스템으로 연결되어있을까? 선언서는 리액티브 프로그래밍을 포용하고 리액티브 스타일로만 코드를 작성하길 바랄까? 음, 사실 그렇지 않다. 선언서는 리액티브 프로그래밍 보다는 *리액티브 시스템*  에 관해 더 많이 이야기 한다.  이러한 것은 같이 섞여 혼란을 야기시킨다.

리액티브 시스템은 특정한 아키텍쳐 설계 원칙을 따른다. 이러한 설계원칙은 오늘날의 요구사항이 증가함에도 불구하고 시스템이 반응적이고, 확장성있고, 장애를 허용하는 시스템으로 이어지기 위한 것이다.

이것을 달성하기 위한 주요 수단은 메세지 전달이다. 리액티브 어플리케이션이 이벤트에 초점을 맞춘 반면에, 리액티브 시스템은 메시지에 초점을 둔다. reactive manifesto는 아래와 같이 차이를 설명한다.

> 메세지는 특정 목적지로 보내지는 데이터의 아이템이다. 이벤트는 주어진 상태가 도달하자마자 컴포넌트에 의해 발생되는 시그널이다. 메세지 기반 시스템에서 주소지정가능한 수신자는 메시지의 도착을 기다리고 그것에 응답한다. 그렇지 않으면 휴면 상태에 있다. 이벤트 기반 시스템에서는 알림 리스너는 이벤트가 발생할때 호출되는 이벤트 소스에 부착된다. 이것은 이벤트 기반 시스템은 주소지정가능한 이벤트 소스에 초점을 맞추는 반면에 메세지 기반 시스템은 주소 지정 가능한 수신자에 초점을 맞춘다.

이러한 메시지에 초점을 맞추면 쉽게 확장할수 있고 위치 투명성으로 이어질 수 있다. 코드의 각각의 조각들이 오직 메시지만을 다룰때 만약 세상의 다른 끝에 있는 기계로 보내나 같은 기계로 메세지를 보내나 그것은 중요하지 않다.

리액티브 시스템은 리액티브 어플리케이션으로 구성될 수 있지만 그럴 필요는 없다. 리액티브 시스템에 대해 리액티브 프로그래밍 스타일을 기본으로하는 것은 좋은 생각일 수 있다. 하지만 이것 없이 리액티브 시스템의 디자인 원칙을 따르는 것 역시 할 수 있다.



## 함수형 리액티브 프로그래밍

결국, 이게 어떻게 함수형 리액티브 프로그래밍과 관련이 있는가? ReactiveX 와 리액티브 프로그래밍과 유사한 라이브러리들은 때때로 "함수형 리액티브 프로그래밍" 이라고 이름 붙여진다. 그러나, 실제로 이건 올바르지 못하다. ReactiveX는 반응형이다 그리고 이것은 맵, 필터 같은 메소드와 익명함수 같은 함수형 프로그래밍으로부터 잘 알려진 많은 요소들을 사용한다. 그러나, 함수형 리액티브 프로그래밍은 완전히 다르게 정의되어 왔다. ReacitveX의 문서에서 말하는 것은 아래와 같다.

> [ReactiveX] 는 가끔 "함수형 리액티브 프로그래밍" 이라고 불린다. 하지만 이것은 잘못된 표현이다. ReacitveX는 함수형, 반응형 일 수 있다. 하지만 "함수형 리액티브 프로그래밍"은 다른 것이다. 가장 큰 차이중 하나는 함수형 리액티브 프로그래밍은 시간이 지남에 따라 계속 변화하는 값에서 동작한다. 반면에 ReactiveX는  시간에 따라 발생하는 이산형 값에서 동작한다.

함수형 리액티브 프로그래밍은 원래  20년도 더 전에 Conal Elliott 과 Paul Hudak의 논문 함수형 리액티브 애니메이션에서 정의 되었다. 그들은 "행동"과 "이벤트" 사이에 차이를 두었다. 이벤트는 기본적으로 우리가 앞에서 봤던 스트림 같은 것이다. 그것들은 시간이 지날때 발생하는 이산형 값이다.

그러나, 행동(Behaviour)은 시간이 지남에 따라 연속적인 값이다. 이것은 미묘하지만 매우 큰 차이다. 차트로 나타내면 아래와 같다.

행동(Behaviour)는 항상 값을 가지고 있다. 이벤트는 가장 최근 발생만을 가지고 있다. 그래서 예를 들자면, 마우스의 위치는 행동이다. 왜냐하면 현재 위치를 요청할 수 있기 때문이다. 그러나 마우스의 클릭은 스트림이다. 마우스  클릭의 값을 요청할 수 없기 때문이다. 오직 가장 최근 발생만을 요청할 수 있다.

ReactiveX는 연속적인 값이 아닌 오직 이산형 이벤트만 모델링 할 수 있다.  그래서 어떻게 빠져나갈 수 있을까?

구별을 그만두는 것은 가능하다. 이벤트의 값을 요청할때, 가장 최근 발생 값을 가질 수 있다. 마우스 이동 같은 연속되는 행동을 가질때, 이것을 샘플링 할 수 있고, 이벤트 스트림처럼 취급할 수 있다. 

그러나, 이것은 문제들을 야기할 수 있다. 예를 들면, 단순히 한 스트림에서 다른 스트림으로 이벤 트를 더함으로써 이벤트 스트림을 결합할 수 있다. 그러나 연속적인 값에게 이것은 말이 되지 않는다. 만약 너가 API에 차이를 두지 않는다면, API가 멍청한 짓을 하는 것을 막을 수 없다.

만약, 왜 구별을 두는게 맞는지에 대해 더 광범위한 토의에 관심이 있다면, 이 [아티클](http://vindum.io/blog/behaviors-and-streams-why-both/)을 추천한다.

두개 다른 셀을 합치는 셀을 가진 워크시트의 예제로 돌아가 보자. 셀들의 값은 행동(behaviour) 들 이다. 그것들은 항상 현재 값을 가지고 있다. 그리고 합을 계산하는 셀의 값은 다른 두개의 셀의 값에 의존한다. 반면에 만약 셀의 값을 변경한다면, 이러한 변경은 이벤트 이다.

만약 지금 워크시트 어플리케이션을 작성하길 원한다면, 2가지 방법을 사용할 수 있다. 합의 값을 업데이트해서 이벤트를 변경하기 위해 콜백을 붙일 수 있다. 그러나 만약 다른 값이 합에 의존한다면, 다른 콜백도 붙여야 한다. 어떻게 이게 그렇게 빨리 되는지 알 수 있다.

반면에, 셀의 값을 행동(behaviour)로써 모델링 할 수 있다. 여기서 합의 값은 그것이 의존하는 다른 두 값의 함수이다. 이건 이 문제를 해결하기 위한 보다 더 자연스러운 접근법으로 보여, 더 우아한 코드를 만들 것이다.
